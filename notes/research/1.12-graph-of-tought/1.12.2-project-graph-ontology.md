**1\. Design goals & namespace**

Goals:

- Represent **projects, goals, tasks, thoughts, agent runs, artifacts, decisions, errors**.
- Make it easy to query:
  - "What tasks are open for project X?"
  - "Which tasks were implemented by which AgentRun?"
  - "Which code artifacts were touched by which task?"
  - "What decisions led to this failing test?"
- Back your **Graph-of-Thought** orchestration: each "thing that happened" becomes an OWL individual.

Let's assume a base namespace:

@prefix : &lt;<http://jido.ai/ontologies/project-graph#>&gt; .

@prefix pg: &lt;<http://jido.ai/ontologies/project-graph#>&gt; .

@prefix xsd: &lt;<http://www.w3.org/2001/XMLSchema#>&gt; .

@prefix rdf: &lt;<http://www.w3.org/1999/02/22-rdf-syntax-ns#>&gt; .

@prefix rdfs: &lt;<http://www.w3.org/2000/01/rdf-schema#>&gt; .

@prefix owl: &lt;<http://www.w3.org/2002/07/owl#>&gt; .

Use pg: or : interchangeably.

**2\. Class model**

**2.1 Core project structure**

- pg:Project
- pg:Goal
- pg:Task
- pg:SubTask (⊑ pg:Task)
- pg:TaskGroup (for grouping related tasks/epics)

**2.2 Reasoning / thought level**

- pg:Thought  
    A reasoning step (could be an LLM message, intermediate explanation, plan refinement, etc.)
- pg:PlanningThought ⊑ pg:Thought
- pg:EvaluationThought ⊑ pg:Thought
- pg:ErrorAnalysisThought ⊑ pg:Thought

**2.3 Execution / agent runs**

- pg:AgentRun  
    A single execution of a CoT/ToT/other runner.
- pg:RunnerType (e.g. CoT, ToT, StaticAnalyzer, TestRunner)
- pg:Outcome
- pg:Decision ⊑ pg:Outcome
- pg:ErrorReport ⊑ pg:Outcome

**2.4 Artifacts**

- pg:Artifact
  - pg:CodeArtifact
    - pg:SourceFile
    - pg:Module
    - pg:Function
  - pg:TestArtifact
  - pg:ConfigArtifact
  - pg:DocumentationArtifact

(You can later align this with your Elixir language ontology.)

**2.5 Status & meta**

You can model status either as classes or individuals of an enumeration-like class. I'd start with:

- pg:Status
  - pg:OpenStatus
  - pg:InProgressStatus
  - pg:CompletedStatus
  - pg:FailedStatus
  - pg:CancelledStatus

And attach a pg:hasStatus property to pg:Goal, pg:Task, pg:AgentRun, etc.

**3\. Object properties (edges)**

Here are the main relations your GoT orchestrator will use.

**3.1 Project & goals/tasks**

- pg:hasGoal
  - domain: pg:Project
  - range: pg:Goal
- pg:hasTask
  - domain: pg:Project
  - range: pg:Task
- pg:goalForProject
  - inverseOf: pg:hasGoal
- pg:taskForProject
  - inverseOf: pg:hasTask
- pg:refinesGoal
  - domain: pg:Task
  - range: pg:Goal

**3.2 Task graph structure**

- pg:dependsOnTask
  - domain: pg:Task
  - range: pg:Task
- pg:precedesTask
  - domain: pg:Task
  - range: pg:Task
- pg:hasSubTask
  - domain: pg:Task
  - range: pg:SubTask

**3.3 Thoughts & runs**

- pg:hasThought
  - domain: pg:Task
  - range: pg:Thought
- pg:generatedByRun
  - domain: pg:Thought
  - range: pg:AgentRun
- pg:hasPrimaryThought
  - domain: pg:AgentRun
  - range: pg:Thought
- pg:appliesToTask
  - domain: pg:AgentRun
  - range: pg:Task
- pg:usesRunnerType
  - domain: pg:AgentRun
  - range: pg:RunnerType

**3.4 Outcomes, decisions, errors**

- pg:hasOutcome
  - domain: pg:AgentRun
  - range: pg:Outcome
- pg:influencesTask
  - domain: pg:Decision
  - range: pg:Task
- pg:reportsErrorFor
  - domain: pg:ErrorReport
  - range: pg:Task

**3.5 Artifacts & code connections**

- pg:producesArtifact
  - domain: pg:Task
  - range: pg:Artifact
- pg:modifiesArtifact
  - domain: pg:Task
  - range: pg:Artifact
- pg:artifactFromRun
  - domain: pg:Artifact
  - range: pg:AgentRun
- pg:touchesModule
  - domain: pg:Artifact
  - range: pg:Module

You can also add cross-ontology links like pg:relatesToConcept later, pointing into your domain-language ontologies.

**3.6 Status**

- pg:hasStatus
  - domain: pg:Project, pg:Goal, pg:Task, pg:AgentRun
  - range: pg:Status

**4\. Data properties (literals)**

Some useful ones:

- pg:hasTitle (xsd:string)
- pg:hasDescription (xsd:string)
- pg:hasIdentifier (xsd:string) - stable ID from JidoCode side
- pg:createdAt (xsd:dateTime)
- pg:updatedAt (xsd:dateTime)

For artifacts:

- pg:filePath (xsd:string)
- pg:moduleName (xsd:string)
- pg:functionName (xsd:string)

For runs:

- pg:startedAt (xsd:dateTime)
- pg:endedAt (xsd:dateTime)
- pg:tokenCount (xsd:integer) - optional, useful for metering

For errors:

- pg:errorMessage (xsd:string)
- pg:errorType (xsd:string)
- pg:errorCode (xsd:string)

**6\. Tiny ABox example**

Just to show how this plays with JidoCode:

@prefix pg: &lt;<http://jido.ai/ontologies/project-graph#>&gt; .

@prefix xsd: &lt;<http://www.w3.org/2001/XMLSchema#>&gt; .

pg:project_jidocode a pg:Project ;

pg:hasIdentifier "jidocode" ;

pg:hasTitle "JidoCode Coding Assistant" .

pg:goal_invoice_aggregate a pg:Goal ;

pg:goalForProject pg:project_jidocode ;

pg:hasIdentifier "goal-123" ;

pg:hasTitle "Add Invoice aggregate with Commanded + Ash" ;

pg:hasStatus pg:OpenStatus .

pg:task_design_invoice a pg:Task ;

pg:taskForProject pg:project_jidocode ;

pg:refinesGoal pg:goal_invoice_aggregate ;

pg:hasIdentifier "task-123-design" ;

pg:hasTitle "Design Invoice events & commands" ;

pg:hasStatus pg:InProgressStatus .

pg:run_001 a pg:AgentRun ;

pg:appliesToTask pg:task_design_invoice ;

pg:usesRunnerType pg:TotRunner ;

pg:startedAt "2025-12-28T12:00:00"^^xsd:dateTime ;

pg:tokenCount 2048 .

pg:TotRunner a pg:RunnerType ;

pg:hasTitle "Tree-of-Thought runner" .

pg:thought_001 a pg:PlanningThought ;

pg:generatedByRun pg:run_001 ;

pg:hasDescription "Explored three candidate event models for Invoice; selected create/pay/cancel." .

pg:decision_001 a pg:Decision ;

pg:hasDescription "Use {InvoiceCreated, InvoicePaid, InvoiceCancelled} events." ;

pg:influencesTask pg:task_design_invoice .

pg:run_001 pg:hasOutcome pg:decision_001 ;

pg:hasPrimaryThought pg:thought_001 .
